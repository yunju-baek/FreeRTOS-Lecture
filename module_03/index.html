Priority, Tick, and Task State

Priority, Tick, and Task State

A task does not simply run or sleep. It moves through scheduler-visible states, and each API call either keeps it eligible for the CPU or deliberately removes it from selection.

What Students Should Be Able To Do

  • Explain Ready, Running, Blocked, and Suspended without relying on memorized diagrams.
  • Predict why a high priority task can starve lower priority work.
  • Explain how the tick interrupt makes delay-based unblocking possible.
  • Choose between vTaskDelay() and vTaskDelayUntil() for periodic work.

FreeRTOS Module 03
Why This Module Matters

Why This Module Matters

For a third-year CS student, this is where OS scheduling concepts meet embedded constraints.

Single CPU Reality

  • Most target boards run many logical tasks on one CPU core.
  • Only one task is Running on that core at a time.
  • Concurrency is created by controlled switching, not by magic parallelism.

Embedded Timing

  • A sensor loop may need a stable release period.
  • A logger should not steal CPU from control work.
  • A delay API affects responsiveness and jitter.

Debugging Impact

  • Wrong priority looks like a missing print.
  • A busy loop looks like a broken scheduler.
  • A tick misunderstanding creates timing bugs.
Priority and state are not vocabulary; they are the grammar of FreeRTOS behavior.

FreeRTOS Module 03
Task State Vocabulary

Task State Vocabulary

State names are shorthand for scheduler eligibility.

Ready

The task can run now if selected. It has all it needs except the CPU.

Running

The task currently owns the CPU core. On a single core, only one task is Running.

Blocked

The task is waiting for time, data, a semaphore, a queue item, or another event.

Suspended

The task is outside normal scheduling until another task explicitly resumes it.

Blocked and Suspended are often confused. Blocked usually has an event or timeout path back to Ready. Suspended has no natural timeout.
State explains whether the scheduler may select the task.

State Transition Overview

State Transition Overview

Reading Order

  • Ready tasks are grouped by priority.
  • The scheduler chooses the highest-priority Ready task.
  • Running tasks become Blocked when they call delay or wait APIs.
  • Tick expiry or an event moves a task back to Ready.
  • Suspended tasks return only through an explicit resume.

FreeRTOS Module 03
The Scheduler Question

The Scheduler Question

At each scheduling point, FreeRTOS asks a small but strict question.

Among the tasks that are Ready, which task has the highest priority?

Not Ready

  • A task blocked on time or data is skipped.
  • A suspended task is skipped.
  • A deleted task no longer participates.

Highest Priority

  • Priority is the first ordering rule.
  • Higher numeric priority means more urgent in FreeRTOS.
  • Priority ties are handled by scheduler policy.

Scheduling Point

  • Tick interrupt
  • Blocking API call
  • Yield
  • ISR unblocking a higher priority task
State filters the candidates; priority orders the candidates.

FreeRTOS Module 03
Priority Numbering

Priority Numbering

FreeRTOS priorities are integer levels from zero to configMAX_PRIORITIES - 1.

Rules

  • Priority 0 is the lowest task priority.
  • The highest valid priority is configMAX_PRIORITIES - 1.
  • Any number of tasks may share the same priority.
  • A priority above the configured maximum is capped by the kernel.
A priority value is a design decision, not just an API argument.

configMAX_PRIORITIES as a Design Limit

The macro sets how many task priority levels the application can use.

configMAX_PRIORITIESuxPrioritytskIDLE_PRIORITY

Too Few

  • Different timing roles are forced into the same level.
  • The design may depend too much on time slicing.
  • Urgency distinctions become less explicit.

Enough

  • Control, communication, logging, and idle work have clear roles.
  • Equal priorities are used intentionally.
  • The priority map can be explained in one table.

Too Many

  • Students may invent artificial levels.
  • Review becomes harder.
  • Some ports have selection cost or implementation limits.
Keep the priority map as small as the design allows.

Tick Interrupt Concept

Tick Interrupt Concept

The tick interrupt gives the kernel a periodic time base.

PeriodicThe hardware timer interrupts at the configured tick rate.
Kernel TimeThe kernel increments its tick count and checks delayed tasks.
UnblockA delayed task can move from Blocked to Ready when its wake tick arrives.
The tick is not the only scheduling trigger, but it is the key trigger for time-based blocking.
Without a time base, delay APIs cannot know when to unblock a task.

configTICK_RATE_HZ and Time Conversion

configTICK_RATE_HZ and Time Conversion

Application code should usually express human-readable time in milliseconds, then convert to ticks.

configTICK_RATE_HZTickType_tpdMS_TO_TICKS()xTaskGetTickCount()

What the macro controls

  • Number of tick interrupts per second.
  • Resolution of time-based delays.
  • Potential overhead from frequent tick interrupts.

What it does not guarantee

  • Sub-tick precision.
  • Exact wake time under all interrupt and priority conditions.
  • A replacement for hardware timers when precise capture/compare timing is needed.
Tick resolution is a system trade-off, not just a constant.

FreeRTOS Module 03
vTaskDelay() as Relative Blocking

vTaskDelay() as Relative Blocking

vTaskDelay() moves the calling task into Blocked state for a number of tick periods relative to the time of the call.

void vTaskDelay( TickType_t xTicksToDelay );vTaskDelay( pdMS_TO_TICKS( 250 ) );

Before Call

  • Task is Running.
  • It owns the CPU.
  • It decides to wait for time.

During Delay

  • Task is Blocked.
  • It is not eligible for CPU selection.
  • Other Ready tasks can run.

After Timeout

  • Task returns to Ready.
  • It does not necessarily run immediately.
  • Priority still decides selection.
Delay is a state transition, not a CPU-consuming sleep loop.

vTaskDelay(1) means the task will unblock after the next tick boundary, not after a guaranteed full tick duration.

vTaskDelayUntil() as Periodic Release

vTaskDelayUntil() is for tasks that need a fixed period rather than a fixed delay after work completes.

void vTaskDelayUntil( TickType_t *pxPreviousWakeTime, TickType_t xTimeIncrement );xLastWakeTime = xTaskGetTickCount();

pxPreviousWakeTime

  • Stores the previous release tick.
  • Updated by the API.
  • Maintains the schedule across loop iterations.

xTimeIncrement

  • The desired period in ticks.
  • Usually produced by pdMS_TO_TICKS().
  • Represents a cadence, not a one-time sleep.

Interpretation

  • vTaskDelay starts counting after work reaches the delay call.
  • Longer work can shift later wake points.
  • vTaskDelayUntil keeps the release schedule anchored.
  • The task still must finish work before the next period expires.

Blocking Is a Scheduling Contract

Blocking Is a Scheduling Contract

A blocking call is a promise to the scheduler: this task cannot make progress until a condition changes.

Time Condition

  • vTaskDelay()
  • vTaskDelayUntil()
  • Timer expiry
  • Tick timeout

Data Condition

  • Queue receive
  • Stream buffer receive
  • Message buffer receive
  • Notification wait

Synchronization Condition

  • Semaphore take
  • Event group wait
  • Mutex acquisition
  • Barrier sync
Module 03 gives the state language used by every later blocking API.

FreeRTOS Module 03
taskYIELD() as Voluntary Reschedule

taskYIELD() as Voluntary Reschedule

taskYIELD() asks the scheduler to run. It does not place the task in Blocked state.

Yield

  • The task remains Ready.
  • It may run again if no equal or higher priority task is a better candidate.
  • It is a cooperation signal, not a timed wait.

Delay

  • The task becomes Blocked.
  • It cannot run until timeout expires.
  • It creates CPU availability for lower priority Ready tasks.
Yield and delay are easy to confuse because both can lead to another task running. The resulting state is different.
Always ask what state the calling task enters after the API call.

Suspended State vs Blocked State

Both states remove a task from CPU selection, but the return path is different.

Blocked has a condition; Suspended has an external controller.

Source Reading: Delay Path

Source Reading: Delay Path

A simplified reading path for delay behavior.

Task loop
  -> vTaskDelay() or vTaskDelayUntil()
  -> calculate wake tick
  -> remove current task from Ready list
  -> insert task into delayed list
  -> scheduler selects another Ready task
  -> tick interrupt advances time
  -> expired delayed task returns to Ready list
This pseudocode is a reading guide. Students should compare it with the actual source names and list operations in the kernel.

Common Misconceptions

Common Misconceptions

These misunderstandings create many early FreeRTOS bugs.

Delay means sleep

  • Better: delay means Blocked state until a tick condition is met.
  • The CPU can run other Ready tasks.
  • The delayed task is not consuming CPU time.

High priority always good

  • A high priority busy loop can starve the system.
  • Priority should represent urgency and blocking behavior.
  • Too many levels make review harder.

Ready means running

  • Ready means eligible.
  • Running means selected.
  • Multiple tasks can be Ready at once.
A misconception slide is useful because it names the bug before it appears in lab.

Design Pattern: Periodic Sensor Task

Design Pattern: Periodic Sensor Task

A periodic task should make its cadence explicit.

void vSensorTask( void *arg )
{
    TickType_t last = xTaskGetTickCount();
    const TickType_t period = pdMS_TO_TICKS( 10 );

    for( ;; )
    {
        read_sensor_sample();
        vTaskDelayUntil( &last, period );
    }
}
The example is about timing intent, not sensor driver details.
Use vTaskDelayUntil() when the loop represents a fixed-rate job.

Review and Board Questions
