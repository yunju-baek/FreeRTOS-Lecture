Priority Inversion, Deadlock, and Gatekeeper Design

Priority Inversion, Deadlock, and Gatekeeper Design

Mutexes make shared resources usable, but they also create ownership, waiting, inversion, and deadlock risks that must be designed and reviewed explicitly.

Course: FreeRTOS Lecture
Embedded Systems Laboratory, Pusan National University
Instructor: Prof. Yunju Baek

Learning path

  • Resource ownership timeline
  • Priority inversion and priority inheritance
  • Deadlock conditions and lock ordering
  • Recursive mutex and reentrancy risk
  • Gatekeeper task as ownership redesign
First target: a lock is a scheduling decision, not just a protection wrapper.

FreeRTOS Module 11
Problem Frame

Problem Frame

Priority inversionA high-priority task waits for a resource owned by a lower-priority task, and medium-priority work can extend the delay.
DeadlockTasks hold resources while waiting for each other, creating a cycle with no progress.
Unbounded hold timeThe owner blocks, prints, allocates, or waits while holding the mutex.
Scattered ownershipMany tasks touch the same resource directly, making ordering and recovery hard to reason about.
Every lock should come with a risk story and evidence plan.

Terminology Before Patterns

Terminology Before Patterns

TermMeaning in this module
Priority inversionA high-priority task is delayed by a lower-priority task that owns a needed resource.
Priority inheritanceA mutex owner temporarily inherits the priority of a higher-priority task waiting for that mutex.
DeadlockA cycle of waiting in which no task can make progress without external recovery.
Hold timeThe time between taking a mutex and releasing it.
Lock orderingA global rule that says which resource must be acquired first when more than one lock is needed.
Recursive mutexA mutex that the same task can take multiple times, requiring matched gives.
Gatekeeper taskA task that exclusively owns a resource and receives serialized requests through a queue.
The vocabulary turns vague lock concerns into reviewable failure modes.

Kernel Book Figure 8.5: Priority Inheritance

Reading Focus

  • The mutex owner temporarily inherits a higher priority.
  • The boost reduces interference from medium-priority work.
  • The boost ends when the owner releases the mutex.
  • Inheritance minimizes inversion but does not fix long hold times.

Source: FreeRTOS Kernel Book, Figure 8.5.

Priority inheritance is mitigation, not permission to hold locks casually.

Deadlock Conditions

Deadlock Conditions

Mutual exclusionAt least one resource can be owned by only one task at a time.
Hold and waitA task holds one resource while waiting for another.
No preemptionThe resource is released only by its owner.
Circular waitA cycle exists: A waits for B while B waits for A.
Breaking any one condition can prevent deadlock, but embedded designs often prefer lock ordering or ownership redesign.
Deadlock is structural; timeouts only make it visible.

FreeRTOS Module 11
Lock Ordering Rule

Lock Ordering Rule

A global lock order avoids circular wait by forcing every task to acquire resources in the same sequence.

Order 1
Take system-state mutex before peripheral-driver mutex.
Order 2
Take UART mutex before log-buffer mutex only if both are required.
Order 3
Release in reverse order and avoid blocking operations while holding either lock.
A lock order that lives only in one developer's memory is not a design rule.
Lock ordering turns deadlock prevention into code-review evidence.

FreeRTOS Module 11
Timeouts and Recovery

Timeouts and Recovery

Timeout value

  • A timed take can reveal unexpected waiting.
  • Timeout value should match system requirements.
  • A timeout is not automatically recovery.

Recovery policy

  • Log owner and resource state.
  • Abort or retry only if resource state remains consistent.
  • Escalate to reset only with clear system policy.

Review warning

  • Infinite waits hide failures.
  • Tiny timeouts can create noise.
  • Retry loops can worsen load.
Timeouts make stuck ownership observable; they do not automatically repair design.

FreeRTOS Module 11
Recursive Mutex as a Narrow Tool

Recursive Mutex as a Narrow Tool

Valid pressure

  • A task re-enters a resource wrapper through controlled call nesting.
  • The same owner must take and give recursively.
  • Every take must have a matching give.

Design smell

  • Recursive mutex used to avoid understanding ownership.
  • Deep call tree hides lock state.
  • Multiple resources still need lock ordering.
Recursive mutexes reduce one reentrancy problem but can hide architecture problems.
Use recursive mutex only when reentrant ownership is intentional.

Shared UART Gatekeeper Case

Shared UART Gatekeeper Case

Problem

  • Several tasks print status.
  • Formatting can take variable time.
  • UART driver should have one owner.

Design

  • Clients send text or structured log records.
  • Gatekeeper owns UART writes.
  • Queue depth gives pressure evidence.

Review

  • Bound message size.
  • Decide drop or block policy.
  • Set gatekeeper priority deliberately.
A UART gatekeeper is often clearer than a print mutex scattered across tasks.

FreeRTOS Module 11
File or Flash Service Case

File or Flash Service Case

Direct lock risk

  • File or flash operations can be long.
  • Client tasks may hold locks while waiting.
  • Power-fail and error recovery complicate ownership.

Gatekeeper fit

  • Single service task owns the storage driver.
  • Requests are serialized.
  • State machine and recovery stay in one place.

Evidence

  • Request queue depth.
  • Service latency.
  • Dropped or rejected request count.
  • Recovery log.
Long services often prefer serialized ownership over shared locks.

FreeRTOS Module 11
Priority Assignment for Gatekeepers

Priority Assignment for Gatekeepers

Too low

  • Requests wait too long.
  • Queue depth grows.
  • Clients may block or drop records.

Too high

  • Service task can steal CPU from application work.
  • Long service operations become latency problems.
  • Backpressure may be hidden until late.

Balanced

  • Priority based on service deadline.
  • Queue size based on burst analysis.
  • Trace confirms response time.
Gatekeeper priority is a design decision, not a default checkbox.

FreeRTOS Module 11
Failure Modes Worth Naming Early

Failure Modes Worth Naming Early

Unbounded hold time

  • Owner blocks or prints while holding a mutex.
  • Higher-priority waiters suffer long delays.
  • Fix by moving slow work outside the protected region.

Circular wait

  • Task A holds X and waits for Y.
  • Task B holds Y and waits for X.
  • Fix by lock ordering or gatekeeper ownership.

Fake ownership

  • Binary semaphore used as a mutex.
  • Priority inheritance is absent.
  • Fix by using mutex for resource ownership.
The failure mode should be predictable from the design diagram.

Testing Checklist

Testing Checklist

Nominal path

  • Resource used by one task at a time.
  • Mutex released on every path.
  • Gatekeeper handles ordinary request load.

Pressure path

  • High-priority task waits for low-priority owner.
  • Two resources requested in different orders.
  • Gatekeeper request burst fills queue.

Failure path

  • Timeout behavior logged.
  • Owner and resource state recorded.
  • Recovery policy documented.
A resource-management lab should intentionally create pressure.

FreeRTOS Module 11
One-Slide Summary

One-Slide Summary

Inversion

  • High priority can wait behind low-priority ownership.
  • Inheritance mitigates medium-priority interference.
  • Hold time still matters.

Deadlock

  • Cyclic waiting is structural.
  • Timeouts reveal but do not automatically repair.
  • Lock order or redesign prevents cycles.

Gatekeeper

  • One task owns the resource.
  • Requests are serialized through a queue.
  • Queue depth becomes evidence.
Resource management is ownership, scheduling, and evidence together.
