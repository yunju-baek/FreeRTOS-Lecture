Semaphores, Mutexes, and Critical Sections

Semaphores, Mutexes, and Critical Sections

Synchronization objects look similar in the API, but they encode different design meanings: event signal, event count, resource ownership, and tiny invariant protection.

Course: FreeRTOS Lecture
Embedded Systems Laboratory, Pusan National University
Instructor: Prof. Yunju Baek

Learning path

  • Binary semaphore as an event signal
  • Counting semaphore as event or resource count
  • Mutex as ownership plus priority inheritance
  • Critical section as very short invariant protection
  • Design review for wrong primitive choices
First target: choose the primitive by meaning, not by similar-looking take/give calls.

Course Position

Course Position

Queues carried data and timers deferred work. Module 10 introduces primitives whose main job is coordination.

SemaphoreA signal or count, usually without payload ownership.
MutexA resource ownership object with priority inheritance behavior.
Critical sectionA very short protected region for invariants that cannot be interrupted.
The module is about selecting the correct coordination meaning.

Problem Frame

Problem Frame

Event happenedA task needs to wake because a condition became true. Payload may not matter.
Event count mattersSeveral identical events or resources must be counted without individual payloads.
Resource has an ownerA peripheral, data structure, or library section needs one task at a time.
Invariant is fragileA few instructions must not be interrupted while shared state changes.
These are four different design problems even though the API names all use take and give vocabulary.
Start with the problem shape before naming the primitive.

Terminology Before APIs

Terminology Before APIs

TermMeaning in this module
Binary semaphoreA synchronization object commonly used as a one-event signal.
Counting semaphoreA semaphore with a count that can represent available resources or accumulated events.
MutexA mutual exclusion object that records ownership and supports priority inheritance.
Critical sectionA short region where interrupts or scheduling are constrained to protect a tiny invariant.
Priority inheritanceA temporary priority boost for a mutex owner when a higher-priority task waits for the same mutex.
OwnershipThe rule that says which task currently controls a shared resource and must release it.
InvariantA condition that must remain true before and after a small shared-state update.
New terms should be tied to the bug they prevent.

Why Binary Semaphore Is Not a Mutex

Why Binary Semaphore Is Not a Mutex

Mutex semantics

  • Represents ownership of a resource.
  • Owner gives the mutex back.
  • Priority inheritance can reduce priority inversion.
  • Review focuses on hold time and owner behavior.
  • Not intended for ISR give/take patterns.

Binary semaphore semantics

  • Represents an event signal.
  • No ownership relationship is recorded.
  • No priority inheritance as a mutex feature.
  • Review focuses on event loss or repeated signal behavior.
  • Can be used with FromISR give variants when appropriate.
Replacing a mutex with a binary semaphore can silently remove priority inheritance.

Priority Inversion Preview

Reading Focus

  • A high-priority task can wait for a low-priority owner.
  • A medium-priority task can extend the delay.
  • The mutex owner is the key actor.

FreeRTOS Module 10
Priority Inheritance Preview

Priority Inheritance Preview

Reading Focus

  • The mutex owner may temporarily inherit a higher priority.
  • The boost ends when the mutex is released.
  • Inheritance minimizes inversion but does not justify long hold times.
  • Only mutex semantics provide this behavior.

Source: FreeRTOS Kernel Book, Figure 8.5.

Priority inheritance is a reason to choose mutex for resource ownership.

Same-Priority Mutex Sequence

  • Equal-priority tasks can still serialize on a mutex.

  • Mutex correctness is separate from fairness intuition.

Critical Section as Tiny Invariant Protection

Critical Section as Tiny Invariant Protection

Invariant example:
A linked-list head pointer and count variable must agree before and after a short update.

Critical section rule

  • Protect only a few instructions.
  • Do not perform I/O, parsing, allocation, or waiting.
  • Understand interrupt latency impact.
  • Prefer mutex when a task-owned resource is being used for longer work.
taskENTER_CRITICAL();
/* update tiny shared invariant */
taskEXIT_CRITICAL();
A critical section is not a general-purpose lock.

Tutorial 21 Reading Path

Critical region

  • Find the protected shared state.
  • Measure how much code is inside.
  • Check for function calls that do too much.

Latency concern

  • Interrupts or scheduling can be constrained.
  • Long work increases response time.
  • Move slow code outside the protected region.

Alternative

  • Use mutex for resource ownership.
  • Use queue or gatekeeper for serialized service.
  • Use atomic operation if available and appropriate.

Mini Case: Buffer Pool

Mini Case: Buffer Pool

Counting semaphore fit

  • N identical buffers available.
  • Taking a buffer decrements availability.
  • Returning a buffer gives the count back.

Queue fit

  • Specific buffer pointers must be passed.
  • Receiver needs identity or metadata.
  • A queue can hold descriptors.

Review evidence

  • Free buffer count.
  • Allocation failure count.
  • Owner of each checked-out buffer.
  • Return path coverage.
A resource count and a resource identity are different facts.

Mini Case: ISR Event Signal

Mini Case: ISR Event Signal

Binary signal

  • One event wakes one processing task.
  • No payload needed.
  • Repeated events may collapse.

Counting signal

  • Repeated events must be counted.
  • No distinct payload per event.
  • Maximum count protects memory.

Queue signal

  • Each event has data.
  • Timestamps or descriptors matter.
  • Queue depth and overflow must be reviewed.
ISR synchronization choice follows the event shape.

FreeRTOS Module 10
Failure Modes Worth Naming Early

Failure Modes Worth Naming Early

Wrong primitive

  • Binary semaphore used as a mutex.
  • Priority inheritance silently absent.
  • Fix by selecting primitive by semantics.

Long critical section

  • Interrupt latency rises.
  • System responsiveness becomes fragile.
  • Fix by shrinking the protected invariant.

Unpaired ownership

  • Mutex taken but not released on error path.
  • Other tasks wait forever or time out.
  • Fix by reviewing every return path.
Synchronization bugs are often naming errors before they are code errors.

Testing Checklist

Testing Checklist

Nominal path

  • Event wakes expected task.
  • Mutex protects expected resource.
  • Critical section preserves invariant.

Pressure path

  • Repeated event burst.
  • Higher-priority task waits for mutex.
  • Interrupt load during protected update.

Failure path

  • Timeout return path checked.
  • Mutex release on error path verified.
  • Critical section kept short.
A synchronization lab should include at least one failure path.

One-Slide Summary

One-Slide Summary

Semaphore

  • Binary: one event signal.
  • Counting: N events or resources.
  • No resource ownership semantics.

Mutex

  • Resource ownership.
  • Priority inheritance behavior.
  • Hold time and release path are central.

Critical section

  • Tiny invariant protection.
  • Latency cost.
  • Not a general lock substitute.
The primitive should communicate the design intent.
