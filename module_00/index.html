Firmware Baseline
Super loop and interrupts

Embedded Firmware Before an RTOS

A small embedded program often begins as one control loop plus a few interrupt handlers.

int main(void)
{
    board_init();

    while (1) {
        read_sensors();
        update_control();
        service_uart();
        refresh_outputs();
        check_buttons();
    }
}

What works at first

  • Low overhead
  • Direct execution order
  • Simple debug path
  • No scheduler configuration

What starts to bend

  • Slow calls delay unrelated work
  • Timing dependencies become hidden
  • Polling wastes CPU time
  • ISR code can grow too much
The issue is not that the loop is wrong. The issue is that timing responsibilities accumulate inside one control path.
3 / 22
Scaling Pressure
Mixed timing requirements

When a Loop Stops Scaling

RTOS pressure appears when different parts of the firmware need different timing behavior.

Fast event
A button, sensor threshold, or packet arrival needs a short response path.
Periodic work
A control loop or sampling routine needs a stable release interval.
Slow work
Logging, formatting, flash access, or network work may run for longer than expected.
Background
Health checks and diagnostics should run when the CPU has slack.

Loop-level symptom

  • The slowest path controls the response time of unrelated work.
  • Interrupt handlers start carrying too much application logic.
  • Adding one feature changes timing behavior elsewhere.
  • Debug output changes timing enough to hide the original bug.
An RTOS gives the firmware a vocabulary for separating work by priority, state, and waiting condition.

Real-Time Vocabulary
Predictability over raw speed

Real-Time Means Predictable, Not Fast

A real-time system is evaluated by whether important work completes within meaningful timing constraints.

Latency

Delay between an event and the start or completion of the response that matters.

Jitter

Variation in timing from one execution to the next, even when the same code path runs.

Deadline

A time boundary after which the result becomes degraded, late, or useless.

Determinism

The ability to reason about timing behavior under a defined set of conditions.

A faster MCU can reduce average execution time. It does not automatically make response time predictable when control flow and priority rules are unclear.
FreeRTOS is mostly a way to describe and enforce execution order, waiting, and wakeup conditions.
5 / 22
Timing Classes
Hard and soft real-time

Hard and Soft Real-Time Requirements

The Kernel Book introduces this distinction before any API because it shapes priority and scheduling decisions.

Hard real-time

  • A missed deadline can mean system failure.
  • Correctness includes timing, not only the computed value.
  • Design tends to reserve priority and bounded paths for the critical work.
  • Example category: protection, emergency response, safety action.

Soft real-time

  • A missed deadline reduces quality or responsiveness.
  • The system may remain usable after late completion.
  • Design can tolerate occasional delay, buffering, or retry.
  • Example category: user input, display update, status reporting.
Priority assignment is not a moral ranking of code. It is a timing decision made for a specific system behavior.

Multitasking
Single-core MCU model

Multitasking on a Single Core

A Cortex-M4F core executes one instruction stream at a time. The RTOS makes multiple execution contexts manageable.

Task AOwn stack, local control flow, waiting points
->
SchedulerReady list, priority rule, blocking state
->
Task BOwn stack, state, and execution history

Not true parallelism

The core still runs one task at a time unless the MCU has multiple cores.

Independent context

Each task has its own stack and scheduler-visible state.

Structured waiting

Tasks can block for time, data, synchronization, or notification.

Multitasking is useful even before parallel hardware appears because it separates timing responsibilities.

Scheduler Preview
Selection and state

What the Scheduler Decides

The scheduler does not run every task. It selects among tasks that are eligible to run.

Preview vocabulary

  • Ready: eligible for CPU selection
  • Running: currently executing on the core
  • Blocked: waiting for time, data, or an event
  • Suspended: outside normal scheduling until resumed
Module 03 turns this preview into a full state transition model.

FreeRTOS Identity
Kernel and libraries

What FreeRTOS Is

FreeRTOS is not a desktop-style operating system. It is a small real-time kernel and a set of modular C libraries for microcontrollers and small microprocessors.

Application codeTasks, drivers, business logic, protocol decisions, product behavior.
FreeRTOS kernelScheduler, task state, queues, semaphores, mutexes, timers, event groups, notifications.
Port and HAL/BSPCortex-M port code, STM32 HAL, board support, interrupt vector, clock and peripheral setup.
ToolchainCompiler, startup file, linker script, generated firmware image, debug probe.

A compact kernel and library ecosystem for deeply embedded software.

Course framing

  • FreeRTOS code is linked into the firmware image.
  • The kernel runs because application code creates tasks and starts the scheduler.
  • Board-specific details remain visible through the port, HAL, BSP, startup, and linker layers.
  • STM32CubeIDE can generate a CMSIS-RTOS V2 wrapper path, but the native FreeRTOS concepts remain the foundation.

Terminology
Task, thread, process

Terminology: Task, Thread, Process

FreeRTOS calls each schedulable execution context a task. That choice avoids desktop OS assumptions.

TermTypical meaningCourse usage
taskFreeRTOS execution unit with its own stack and scheduler-visible state.The main term for Module 02 onward.
threadGeneral OS term for a schedulable execution path inside a process.Used only for analogy when needed.
processUsually implies an address space and OS-managed resource boundary.Usually avoided for MCU FreeRTOS explanations.
kernel objectQueue, semaphore, mutex, timer, event group, notification, buffer.Used to group FreeRTOS APIs by design purpose.
A task is not a separate program. On a typical Cortex-M FreeRTOS system, tasks share the same address space.

Motivation
Why use an RTOS

Why Use an RTOS?

The Kernel Book frames RTOS use as a set of software design benefits, not only a way to chase deadlines.

Timing abstraction

Delay, timeout, periodic release, and event wait become explicit API-level concepts.

Maintainability

Timing details move out of tangled module-to-module control flow.

Modularity

Tasks become independent modules with defined responsibility and interface.

Testing and reuse

Smaller independent execution paths are easier to test and move to another project.

Efficiency

Blocked tasks do not poll. They wake when time, data, or events arrive.

Idle time

The idle task exposes spare CPU capacity and low-power opportunities.

Interrupt discipline

ISRs can stay short by deferring processing to task context.

Design Shift
Event-driven firmware

Event-Driven Firmware

A major RTOS shift is moving from repeated checking to blocked waiting and explicit wakeup.

Polling-heavy loop

  • Repeated checks even when nothing changed
  • CPU time spent asking the same question
  • Long loop path increases response variation
  • State scattered across flags and timing counters

RTOS event-driven path

  • Task blocks until data, time, or event is ready
  • Scheduler runs another Ready task while waiting
  • Queue, semaphore, notification, or event group encodes the wakeup reason
  • Timeout policy becomes part of the design
Module 06 through Module 14 are mostly a study of different wakeup and communication objects.
13 / 22
Interrupt Discipline
Deferred processing preview

Interrupts Stay Short

An RTOS does not remove interrupts. It gives the firmware a disciplined way to leave heavy work outside ISR context.

Preview rule

  • ISR handles the immediate hardware event.
  • ISR uses a FromISR API when FreeRTOS interaction is needed.
  • A task completes longer processing in normal task context.
  • The ISR may request a context switch if a higher priority task becomes Ready.
Module 08 will make this a concrete API and priority design problem.

Feature Map
Problem to object

FreeRTOS Kernel Feature Map

Module 00 only names the tools. Later modules explain mechanisms, API details, failure modes, and STM32 checkpoints.

Task

Independent execution context and responsibility boundary.

Scheduler

Priority and state based CPU ownership decision.

Queue

Data transfer plus blocking behavior between tasks.

Mutex

Shared resource ownership with priority inheritance.

Timer

Time event callback without application polling loops.

Event group

Multiple conditions encoded as waitable bits.

Notification

Lightweight direct signal to one task.

Buffers

Byte stream and message boundary data paths.

Object choice is a design decision about ownership, waiting, data shape, and timing.

License and Family
FreeRTOS, OpenRTOS, SafeRTOS

FreeRTOS, OpenRTOS, and SafeRTOS

The course uses open-source FreeRTOS, but the family context matters for industry discussions.

FreeRTOS

  • MIT open-source license
  • Commercial application use allowed
  • Users retain their own intellectual property
  • Main target for this course

OpenRTOS

  • Commercially licensed version
  • Provided through a third party under license
  • Useful context for legal and procurement paths
  • Not the lab target here

SafeRTOS

  • Related usage model
  • Developed for safety standard compliance claims
  • Relevant to safety-critical product domains
  • Not a shortcut to safety certification
The practical lesson: FreeRTOS learning maps into a larger commercial and safety-aware ecosystem.

Module 00 Self Lab

Project Boundary Check

This checkpoint is deliberately light. The goal is conceptual readiness before writing FreeRTOS code.

Board and environment

  1. Open STM32CubeIDE and confirm the NUCLEO-F439ZI board or STM32F439ZI MCU path.
  2. List five board features visible in the ST board description.
  3. Mark which features are needed for core FreeRTOS labs.
  4. Mark which features should stay as future extension topics.

RTOS decision sketch

  1. Invent a small firmware scenario with at least four activities.
  2. Classify each activity as periodic, event-driven, continuous, or background.
  3. Identify one timing risk in a super loop version.
  4. Choose which activity would deserve the highest task priority and explain why.
Checkpoint output: a one-page note with workload categories, one timing risk, and a first task-priority hypothesis.
