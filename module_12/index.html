Event Groups and Barrier Synchronization

Event Groups and Barrier Synchronization

Event groups combine multiple boolean conditions into one waitable kernel object. The hard part is not setting a bit, but defining bit meaning, wait policy, and clear policy.

Course: FreeRTOS Lecture
Embedded Systems Laboratory, Pusan National University
Instructor: Prof. Yunju Baek

Learning path

  • EventBits_t, bit numbers, and named masks
  • Wait-any versus wait-all behavior
  • Clear-on-exit and bit lifetime
  • Barrier synchronization with xEventGroupSync
  • Object-choice review against queues and notifications
First target: treat each bit as a named condition with an explicit lifetime.

Problem Frame

Problem Frame

Status setSeveral subsystems become ready independently, and one task needs to observe the combined status.
Wait-anyA task should wake when at least one of several conditions becomes true.
Wait-allA task should wake only after a full set of conditions becomes true.
BarrierSeveral tasks reach a synchronization point and should proceed only when all have arrived.
The design question is the wait condition, not the API call.

Terminology Before APIs

TermMeaning in this module

Event Group Versus Queue

Decision Filter

  • Use a notification when one task is the direct receiver.

Mini Case: System Startup Readiness

Mini Case: System Startup Readiness

Bits

  • NET_READY_BIT.
  • SENSOR_READY_BIT.
  • STORAGE_READY_BIT.
  • CLOCK_READY_BIT.

Wait policy

  • Coordinator waits for all ready bits.
  • Timeout leads to degraded mode or startup failure.
  • Returned bits document missing subsystem.

Clear policy

  • Ready bits may remain set while subsystem is ready.
  • Failure bits may clear only through owner task.
  • Policy should be written next to mask definitions.
Startup readiness is a natural wait-all design.

FreeRTOS Module 12
Failure Modes Worth Naming Early

Failure Modes Worth Naming Early

Unnamed bits

  • Magic constants hide condition meaning.
  • Bit reuse creates conflicting interpretations.
  • Fix with named masks and a bit table.

Wrong clear policy

  • Bits disappear before another task observes them.
  • Stale bits wake tasks repeatedly.
  • Fix by documenting clear ownership.

Payload expectation

  • A bit says that something happened but carries no data.
  • Ordering and history are not preserved.
  • Fix with queue, stream, message buffer, or notification value.
Event group bugs often start as unclear bit lifetime.

Testing Checklist

Nominal path

  • Set one bit and verify wait-any behavior.
  • Set full mask and verify wait-all behavior.
  • Run barrier with all participants arriving.

Pressure path

  • Delay one participant.

  • Repeat barrier sequence.

Failure path

  • Wrong clear policy is visible.
  • Payload requirement triggers object redesign.
