FreeRTOS Module 05
Memory Management, Heap, and Stack

Memory Management, Heap, and Stack

Every FreeRTOS object needs storage. This module separates where that storage comes from, how stacks fail, and why allocation policy is a reliability decision.

The goal is not to memorize every heap implementation, but to read memory behavior from source, configuration, and runtime evidence.

Module Contract

  • Separate kernel object storage, task stacks, application buffers, and the FreeRTOS heap.
  • Compare dynamic and static allocation as system design choices.
  • Explain heap_1 through heap_5 in terms of allocation, free, coalescing, and memory regions.
  • Use stack high-water mark, malloc failed hook, and stack overflow hook as diagnostic evidence.
01
FreeRTOS Module 05
Course Position

Course Position

Scheduling explains who runs next. Memory management explains whether the objects needed for that schedule can exist safely for the life of the system.

Kernel objectsTasks, queues, semaphores, timers, and event groups have internal control blocks.
Allocation policyThe heap implementation decides how dynamic kernel memory is obtained and reused.
Stack marginEach task stack is separate and must survive worst-case call depth and interrupt interaction.
Memory is the hidden contract behind every task and kernel object.

FreeRTOS Module 05
Why Memory Is a Real-Time Topic

Why Memory Is a Real-Time Topic

Memory errors in embedded systems often look like timing, scheduling, or driver bugs. A reliable FreeRTOS design makes memory behavior explainable.

Startup success

  • Can all required objects be created?
  • Does the heap have enough space for dynamic creation?
  • Are static buffers sized and linked correctly?

Runtime stability

  • Does allocation continue after startup?
  • Can fragmentation accumulate?
  • Are stack margins still safe under peak workload?

Failure evidence

  • Which hook should fire?
  • Which counter or high-water mark should be inspected?
  • Which object creation call returned failure?
Treat memory policy as part of system reliability, not just resource accounting.

FreeRTOS Module 05
Memory Areas in a FreeRTOS Firmware Image

Memory Areas in a FreeRTOS Firmware Image

A single firmware image contains several storage areas with different lifetimes and ownership rules.

Code / flash
Program instructions and constant data. Usually not allocated at runtime.
Global / static
Application globals, static buffers, and statically allocated FreeRTOS object storage.
FreeRTOS heap
Storage used by pvPortMalloc() for dynamic tasks, queues, timers, semaphores, and application calls if used.
Task stacks
Separate stacks for each task. Often allocated from the FreeRTOS heap when tasks are created dynamically.
C runtime stack
Startup and interrupt context may have separate stack rules depending on the port and MCU.

Teaching Point

  • Do not say simply 'memory is low' without naming the region.
  • A task stack problem and a heap exhaustion problem require different evidence.
  • Static allocation moves responsibility from the heap implementation to application-provided buffers.
  • On STM32, the linker script and startup file also shape the real memory map.
The first debugging question is which memory area is failing.

Dynamic Task Creation Storage

Dynamic Task Creation Storage

xTaskCreate() dynamically creates at least a TCB and a task stack. The exact allocation path is implemented in the kernel and selected heap file.

xTaskCreate()TCB_tStackType_tpvPortMalloc()configTOTAL_HEAP_SIZE

What students should trace

  • The stack depth argument is in words, not bytes on many ports.
  • The TCB and stack are separate concepts even when allocated together.
  • Creation returns a status that must be checked.

What students should not assume

  • That a created task has infinite stack.
  • That heap exhaustion is obvious in logs.
  • That increasing heap size fixes stack overflow.

Static Task Creation Storage

Static Task Creation Storage

xTaskCreateStatic() moves storage responsibility to the application. The kernel still manages the task, but it does not allocate the TCB and stack from the FreeRTOS heap.

static StaticTask_t sensorTcb;
static StackType_t sensorStack[ SENSOR_STACK_WORDS ];

xTaskCreateStatic(
    vSensorTask,
    "Sensor",
    SENSOR_STACK_WORDS,
    NULL,
    SENSOR_PRIORITY,
    sensorStack,
    &sensorTcb );
Static allocation makes storage visible in the source file and linker map.

Dynamic vs Static Allocation

Dynamic vs Static Allocation

The choice is not about which API looks nicer. It is about who owns memory sizing and when failures can occur.

QuestionDynamic allocationStatic allocation
Who supplies storage?FreeRTOS heap via pvPortMalloc()Application buffers provided to creation API
When can failure occur?At creation time if heap is insufficient or fragmentedAt build/link/design review time if buffers are wrong
Runtime flexibilityBetter for optional or created-on-demand objectsBetter for fixed, safety-focused object sets
Evidence to inspectHeap free space, failed allocation hook, returned statusBuffer declarations, map file, stack high-water marks
Dynamic and static allocation are different reliability contracts.

Configuration Gates for Allocation

Configuration Gates for Allocation

FreeRTOSConfig.h can allow dynamic allocation, static allocation, or both.

configSUPPORT_DYNAMIC_ALLOCATIONconfigSUPPORT_STATIC_ALLOCATIONconfigTOTAL_HEAP_SIZEconfigAPPLICATION_ALLOCATED_HEAP

Dynamic enabled

  • Creation APIs that allocate from heap are available.
  • Heap implementation must be selected in the build.
  • Creation status must be checked.

Static enabled

  • Static creation APIs are available.
  • Application provides storage objects.
  • Object memory becomes visible in source review.

Both enabled

  • Common in teaching examples.
  • Useful for comparing policies.
  • Requires clear local coding convention.
Config macros should express an allocation policy, not just compile switches.

FreeRTOS Module 05
FreeRTOS Heap Implementations at a Glance

FreeRTOS Heap Implementations at a Glance

The heap implementation is selected by compiling exactly one heap_n.c file into the project.

heap_1No free. Smallest mental model. Good when all dynamic objects are created once.
heap_2Allows free but does not coalesce adjacent free blocks. Fragmentation risk remains high.
heap_3Wraps the C library malloc/free. Behavior depends on the C runtime implementation.
heap_4Allows free and coalesces adjacent free blocks. Common general-purpose choice.
heap_5Like heap_4 but supports multiple non-contiguous memory regions.
The heap file is a policy module; compile one that matches the system behavior.

FreeRTOS Module 05
heap_1: Allocate Once

heap_1: Allocate Once

heap_1.c is intentionally limited: it allocates but never frees.

Strength

  • Simple code path.
  • No fragmentation from free/reuse.
  • Predictable when all objects are created before scheduler start.

Limit

  • Deleted objects do not return memory.
  • Repeated create/delete patterns are a poor fit.
  • Dynamic lifetime changes are not supported.

Teaching use

  • Good first model.
  • Good for startup-only systems.
  • Easy to connect to Figure 3.1.
heap_1 trades flexibility for explainability.
14
FreeRTOS Module 05
heap_2: Free Without Coalescing

heap_2: Free Without Coalescing

heap_2.c supports freeing memory, but it does not merge adjacent free blocks into larger blocks.

What it can do

  • Reuse a freed block if a later allocation fits.
  • Support some create/delete patterns.
  • Expose fragmentation as a teaching topic.

What it cannot do well

  • Recover large contiguous blocks from neighboring small free blocks.
  • Guarantee long-running behavior under varied allocation sizes.
  • Make arbitrary dynamic lifetimes safe.
Students should learn the word coalescing here; it explains why total free bytes can be misleading.
Free space is not the same as allocatable contiguous space.
15
FreeRTOS Module 05
heap_4: Free With Coalescing

heap_4: Free With Coalescing

heap_4.c supports free and coalesces adjacent free blocks, reducing fragmentation risk for many systems.

Mechanism

  • Keeps a free block list.
  • Splits blocks for allocations.
  • Merges adjacent blocks when memory is freed.

Practical use

  • Common FreeRTOS heap choice.
  • Useful when objects can be deleted.
  • Still requires bounded design discipline.

Review question

  • Are runtime allocation sizes bounded?
  • Are delete/recreate paths rare?
  • Is failure handling explicit?
Coalescing helps, but it does not remove the need for allocation policy.

FreeRTOS Module 05
heap_5: Multiple Memory Regions

heap_5: Multiple Memory Regions

heap_5.c extends heap_4-style behavior to multiple non-contiguous memory regions.

Why it exists

  • Some MCUs expose RAM in separated banks.
  • External RAM may be present beside internal SRAM.
  • A single contiguous heap array may not represent the hardware well.

Design caution

  • Region order matters.
  • Memory speed and DMA accessibility may differ.
  • Not every object should necessarily live in any region.
HeapRegion_tvPortDefineHeapRegions()heap_5.c
heap_5 is about hardware memory topology, not just more heap.

FreeRTOS Module 05
pvPortMalloc() and vPortFree()

pvPortMalloc() and vPortFree()

FreeRTOS heap files implement the allocation functions used by the kernel and optionally by application code.

pvPortMalloc()vPortFree()xPortGetFreeHeapSize()xPortGetMinimumEverFreeHeapSize()

Allocation

  • Returns a pointer or NULL.
  • May align the block for the port.
  • Consumes bytes plus allocator metadata.

Freeing

  • Only supported in heap implementations that implement free.
  • Returns a block to the allocator.
  • May coalesce depending on heap file.

Observation

  • Free heap is not the whole story.
  • Minimum-ever free heap helps size margins.
  • Fragmentation needs policy awareness.
Allocation APIs are kernel infrastructure; application use should be deliberate.

FreeRTOS Module 05
Task Stack Sizing

Task Stack Sizing

The stack depth passed to task creation must account for worst-case local variables, call depth, library calls, and port context.

Direct usage

  • Local arrays
  • Function call frames
  • Nested helper calls
  • Printf-like formatting

Indirect usage

  • Driver library calls
  • Newlib or C library behavior
  • Compiler optimization changes
  • Floating-point context on some ports

Review habit

  • Start with a reasoned estimate.
  • Measure high-water mark.
  • Keep margin for worst-case paths.
  • Document why the size is acceptable.
Stack size should be justified by evidence, not copied from an example.

FreeRTOS Module 05
High-Water Mark Diagnostics

High-Water Mark Diagnostics

A high-water mark reports the minimum remaining stack space observed for a task since it started.

uxTaskGetStackHighWaterMark()uxTaskGetStackHighWaterMark2()configCHECK_FOR_STACK_OVERFLOW

Use it for

  • Checking whether a task has too much or too little stack.
  • Comparing normal workload and stress workload.
  • Documenting stack margin after changes.

Limitations

  • It only reports observed history, not future worst case.
  • Units are stack words or bytes depending on API and port details.
  • Instrumentation itself should be controlled.
High-water mark is a margin measurement, not a mathematical proof.

Stack Overflow Hook

When enabled, stack overflow checking gives a last-resort diagnostic path for stack corruption risk.

void vApplicationStackOverflowHook(
    TaskHandle_t xTask,
    char *pcTaskName )
{
    /* Stop, log, or break into debugger. */

}
The hook should not be treated as normal recovery. It is evidence that the design or workload exceeded a safety margin.
A stack overflow hook is a diagnostic alarm, not a control-flow feature.

FreeRTOS Module 05
Malloc Failed Hook

Malloc Failed Hook

When dynamic allocation fails, the malloc failed hook can give a controlled breakpoint or logging path.

void vApplicationMallocFailedHook( void )
{
    /* pvPortMalloc() returned NULL. */
    taskDISABLE_INTERRUPTS();
    for( ;; );
}

Questions to ask

  • Which object was being created?
  • How much free heap was available before the call?
  • Was this allocation expected after startup?

Do not hide it

  • Do not continue silently.
  • Do not keep retrying without a policy.
  • Do not assume a larger heap is the correct fix.
Allocation failure should become visible during development and test.

FreeRTOS Module 05
Fragmentation Scenario

Fragmentation Scenario

Fragmentation means the allocator has free memory, but not in a useful shape for the next request.

Step 1
Allocate A, B, and C from a contiguous heap region.
Step 2
Free B. Total free memory increases, but the free block is between A and C.
Step 3
Request D larger than B. The allocation may fail unless a sufficiently large contiguous block exists.
Step 4
If adjacent free blocks are coalesced, later allocation opportunities improve.
Fragmentation is about shape and lifetime, not just byte count.

FreeRTOS Module 05
Allocation Lifetime Policy

Allocation Lifetime Policy

A good embedded design can often describe when allocation is allowed.

Startup-only

  • Create all tasks and kernel objects before scheduler start.
  • Runtime behavior has no heap churn.
  • Fits heap_1 or static allocation well.

Mode-based

  • Create optional objects when entering a feature mode.
  • Delete on exit only if lifetime is rare and bounded.
  • Requires failure path and testing.

Unbounded runtime

  • Frequent variable-size allocation.
  • Hard to prove long-term behavior.
  • Usually inappropriate for core real-time paths.
The safest dynamic allocation is often the allocation that happens before real-time work begins.

FreeRTOS Module 05
Design Pattern: No Runtime Allocation in Control Path

Design Pattern: No Runtime Allocation in Control Path

A common embedded rule is to avoid allocation in the highest-criticality runtime path.

Before scheduler

  • Create tasks
  • Create queues
  • Create timers
  • Verify handles and counters

During runtime

  • Use pre-created objects
  • Pass data through bounded queues or buffers
  • Measure stack and heap margins

At failure

  • Fail visibly
  • Preserve diagnostic evidence
  • Avoid silent fallback that hides memory pressure
Bounded runtime behavior is easier to explain and test.
