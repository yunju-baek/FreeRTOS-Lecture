Queue Extension Patterns and ISR Queues

Queue Extension Patterns and ISR Queues

A queue can carry pointers, represent a mailbox, join a queue set, or cross an interrupt boundary. Each extension changes the design contract.

The module is written for third-year computer science students: new terms are defined before use, source files are named, and visuals are used as reasoning anchors.

Module Contract

  • Explain pointer queue ownership and lifetime rules.
  • Use mailbox and queue set patterns only when their semantics fit the problem.
  • Distinguish task-context queue APIs from FromISR queue APIs.
  • Trace Tutorial 09 and Tutorial 10 as task-to-task and ISR-to-task queue cases.

FreeRTOS Module 07
Problem Frame

Problem Frame

The practical pressure behind this module can be separated into three design questions.

Large data

  • Copying large structs is expensive.
  • Pointer queues reduce copy cost.
  • Pointer lifetime becomes the real risk.

Many sources

  • A worker may need input from several queues or semaphores.
  • Queue sets can multiplex readiness.
  • Complexity can rise quickly.

Interrupt events

  • ISR cannot block.
  • The correct API suffix matters.
  • The awakened task may need an immediate context switch.
Start with the system problem, then introduce the kernel mechanism.

Terminology Before APIs

Terminology Before APIs

These words appear repeatedly in the source, API names, and design discussions.

TermMeaning in this module
Pointer queueA queue whose item is a pointer value. The pointed buffer is not copied.
OwnershipThe rule that says which task may read, modify, or free a buffer at each point.
MailboxA latest-value pattern, often implemented with a length-one queue and overwrite semantics.
Queue setA FreeRTOS object that lets a task wait on multiple queues or semaphores.
FromISR APIAn API variant designed for interrupt context; it does not block.
xHigherPriorityTaskWokenA flag that records whether an ISR unblocked a higher-priority task.
Terminology should describe behavior, ownership, or caller context.

FreeRTOS Module 07
Failure Modes

Failure Modes

The most useful slide in a lecture is often the one that names the bug early.

Dangling pointer

  • Queueing an address of a local variable.
  • Receiver uses data after sender stack changed.
  • Fix with static buffer, heap buffer, or buffer pool ownership.

Wrong API context

  • Calling task-context API in ISR.
  • Passing a block time from interrupt context.
  • Fix with FromISR API family.

Overdesigned multiplexing

  • Queue set used when one input queue would be clearer.
  • Harder debugging and ownership tracing.
  • Fix by simplifying object choice.
Predictable failure modes should become checklist items.

FreeRTOS Module 07
API Pitfalls

API Pitfalls

The names differ by module, but the mistakes repeat.

Wrong context

  • Blocking call in ISR.
  • Long work inside service context.
  • Ignoring priority boundary.

Wrong semantics

  • Using a data object as a signal.
  • Using a signal object as a lock.
  • Expecting payload where none exists.

Wrong evidence

  • Changing code before measuring.
  • Trusting one log line.
  • Ignoring return values.
Pitfalls become less mysterious when they are grouped by context, semantics, and evidence.
