Module 02 · Task Definition
A thread-like execution unit inside firmware

What Is a FreeRTOS Task?

A FreeRTOS task is a schedulable execution unit. It is similar to a lightweight thread: it has its own stack and execution point, but normally shares the same address space with other tasks.

Function body

The C code that implements the work. A task function is where application behavior is written.

Kernel metadata

The kernel needs state, priority, stack pointer, and list links to schedule the task.

Runtime stack

Each task needs its own call stack so local variables and nested calls are separate.

The task analogy to a thread is useful, but do not assume desktop-process protection or virtual memory.
A task combines application code with kernel-managed runtime state.

Module 02 · Task Function
The C entry point for a task

Task Function Prototype

FreeRTOS task functions use a fixed prototype. The parameter is a generic pointer so the creator can pass task-specific data without changing the task function signature.

void TaskFunction( void *pvParameters )
{
    for( ;; )
    {
        /* Task work goes here. */
    }
}

Read the signature

  • void return type: task should not return
  • void * parameter: generic data handoff
  • Function name becomes the entry point stored by the kernel
  • Actual scheduling begins after the scheduler starts
The task function is application code, but it becomes schedulable only after task creation.

Module 02 · Task Loop
Why tasks usually do not return

Infinite Loop and Blocking Point

Most task functions run forever, but not by busy-waiting forever. A well-behaved task does useful work and then blocks, delays, or waits so other tasks can run.

RunExecute one unit of task work
→
WaitvTaskDelay(), queue receive, semaphore take
→
Ready againScheduler can select it later

Good loop

  • Uses blocking APIs or delay
  • Lets scheduler run other Ready tasks
  • Has a clear period or event source

Risky loop

  • Busy loop without blocking
  • Consumes CPU continuously
  • Can starve equal or lower-priority tasks
A task loop should usually contain a point where the task gives the scheduler another choice.

Module 02 · Task Anatomy
The pieces created around a task function

Task Object Anatomy

A task is easier to understand as a bundle: function code, TCB, stack, priority, state, and optional handle. The function alone is not enough for scheduling.

Read the visual

  • The kernel adds metadata such as state and priority

The task function supplies behavior; the task object supplies schedulability.

Module 02 · TCB
Kernel metadata for one task

Task Control Block

A Task Control Block, or TCB, is kernel-owned metadata for a task. You normally do not edit it directly, but many runtime symptoms are explained by what the kernel stores there.

Scheduling fields

  • Current state
  • Priority
  • List links used by scheduler

Stack fields

  • Saved stack pointer
  • Stack bounds or high-water data when enabled
  • Context switch bookkeeping

Debugging fields

  • Task name
  • Task handle relation
  • Trace and diagnostics support
The exact internal structure is kernel implementation detail, but the concept is stable: the TCB is where the kernel remembers the task.
A task without a TCB cannot be managed by the scheduler.

Module 02 · Stack
Each task needs its own call frames

Private Stack per Task

Each task needs a stack for local variables, return addresses, saved CPU context, and nested function calls. This is why creating a task is also a memory-allocation decision.

Task A stack
Local variables and call frames for Task A
Task B stack
Independent call frames for Task B
TCB records
Kernel metadata points to each saved stack location

Common mistake

  • Stack depth is not a random large number
  • Too small: overflow or corrupted behavior
  • Too large: wasted RAM on small MCUs
  • Unit depends on FreeRTOS port conventions
Stack size is a design parameter, not just an API argument.

Module 02 · Dynamic Creation
The first task creation API

xTaskCreate() API Shape

xTaskCreate() creates a task using FreeRTOS heap memory. The API receives the task behavior, metadata choices, stack depth, priority, and an optional handle output.

BaseType_t xTaskCreate(
    TaskFunction_t pxTaskCode,
    const char * const pcName,
    const configSTACK_DEPTH_TYPE usStackDepth,
    void * const pvParameters,
    UBaseType_t uxPriority,
    TaskHandle_t * const pxCreatedTask
);
The return value tells whether the kernel successfully allocated and registered the task.
Read the API as a constructor for a task object.

Module 02 · Parameters I
Function, name, and parameter handoff

Task Code, Name, and Parameters

The first group of parameters tells the kernel where the task starts, how to label it, and what initial data pointer should be passed to the task function.

pxTaskCodeFunction pointer to the task function. In Tutorial 02, this is prvTaskFunction.
pcNameHuman-readable task name. Useful for debugging and trace tools, not for scheduling decisions.
pvParametersGeneric pointer passed to the task function. Often NULL in simple tutorials.
If pvParameters points to data, the data must remain valid after the task starts. Lifetime mistakes are common.
The parameter pointer is powerful because it is generic, and risky for the same reason.

Module 02 · Dynamic Memory Path
Where TCB and stack memory come from

Dynamic Task Creation Path

With xTaskCreate(), the kernel allocates memory for the TCB and stack from the FreeRTOS heap. The task can be created only if the allocation succeeds.

FreeRTOS heap

The kernel allocates memory for the TCB and stack.

Task metadata

The task becomes visible to scheduler data structures.

Ready path

A successful task can later be selected by the scheduler.

Convenience

  • Shorter application code
  • Memory request expressed by stack depth
  • Good for tutorials and flexible prototypes

Responsibility

  • Heap must be configured
  • Allocation can fail
  • Failure path must be checked
Dynamic creation trades shorter application code for heap dependency.

Module 02 · Return Value
Detecting creation failure early

BaseType_t, pdPASS, and configASSERT()

Dynamic creation can fail because memory allocation can fail. The return value gives application code a chance to detect that failure immediately.

BaseType_t

FreeRTOS integer type commonly used for success/failure returns and small status values.

pdPASS

Expected success value for many FreeRTOS API calls that return a status.

configASSERT()

Debug-time check that stops execution when an assumption is violated.

Ignoring the return value can hide the real problem until the scheduler starts and nothing useful runs.
The first debugging checkpoint for task creation is the API return value.

Module 02 · Scheduler Start
The execution boundary in main

Before and After vTaskStartScheduler()

Before scheduler start, main() follows ordinary C control flow. After scheduler start, Ready tasks become the execution units selected by the scheduler.

Read the visual

  • Before the boundary, setup code creates task objects
  • The scheduler boundary changes the execution model
  • After the boundary, Ready tasks and Idle task share CPU time
  • The tick becomes part of timing behavior

Expected path

The scheduler starts, picks a Ready task, and the task function begins running according to scheduling rules.

Failure path

If scheduler start returns, the system usually failed to create required kernel resources, often because memory is insufficient.

Scheduler start is the boundary between setup code and task-based execution.

Module 02 · Delay Preview
Why the tutorial task does not busy-loop

vTaskDelay() and Tick Conversion

Tutorial 02 prints a message and then delays. This delay moves the task out of Running state for a period, allowing other tasks or the idle task to run.

for( ;; )
{
    fprintf( stderr, "Tutorial 2 running...\\r\\n" );

    vTaskDelay( pdMS_TO_TICKS( 1000 ) );
}
vTaskDelay() moves the task out of Running state for a tick-based interval; pdMS_TO_TICKS() keeps the source readable in milliseconds.
A task loop should usually include a scheduler-visible waiting point.

Module 02 · Static Creation
Avoiding heap allocation for task memory

Why Static Task Creation Exists

Static creation moves TCB and stack memory ownership from the FreeRTOS heap to application-provided buffers. This can make memory use more explicit and predictable.

Dynamic style

  • Application requests memory through API
  • Kernel allocates TCB and stack from heap
  • Shorter code, but allocation can fail

Static style

  • Application declares TCB buffer
  • Application declares stack buffer
  • Kernel uses the provided memory

Design reason

  • Clear memory ownership
  • Less runtime allocation uncertainty
  • Common in stricter embedded styles
Static creation is more verbose because memory ownership is explicit.

Module 02 · Static API
The same task, different memory ownership

xTaskCreateStatic() API Shape

The static API adds two important arguments: a stack buffer and a TCB buffer supplied by the application.

TaskHandle_t xTaskCreateStatic(
    TaskFunction_t pxTaskCode,
    const char * const pcName,
    const uint32_t ulStackDepth,
    void * const pvParameters,
    UBaseType_t uxPriority,
    StackType_t * const puxStackBuffer,
    StaticTask_t * const pxTaskBuffer
);
The return value is a task handle. A NULL return means the task was not created.
Static creation asks the application to bring the memory before calling the kernel.

Module 02 · Stack And Parameters
Two common beginner mistakes

Stack Depth and Parameter Lifetime

Two mistakes appear early in task creation: treating stack depth as an arbitrary byte count, and passing a pointer to data that does not live long enough.

Stack depth caveat

  • FreeRTOS stack depth is expressed in stack units for the port, not always raw bytes
  • Too small can corrupt execution
  • Too large wastes RAM
  • Use diagnostics and high-water checks when available

Parameter lifetime

  • pvParameters is only a pointer
  • Do not pass a pointer to a local variable that disappears
  • Prefer static, global, heap, or clearly owned data
  • Document who owns the pointed-to data
The API accepts numbers and pointers; the design must supply valid memory assumptions.

Module 02 · Design Choice
Dynamic and static task creation compared

Dynamic vs Static Task Creation

Neither style is universally better. The choice depends on memory ownership, failure policy, and how predictable the firmware must be.

QuestionDynamic creationStatic creation
Who provides memory?FreeRTOS heapApplication buffers
Code lengthShorter creation codeMore setup code
Failure modeHeap allocation can failBuffer pointer or size mistakes can fail
Design styleConvenient and flexibleExplicit and predictable
If the system must avoid runtime heap allocation, which creation style better matches that policy?
Memory ownership is the main difference, not the task function itself.

Module 02 · Debugging
Where task creation fails

Debugging Task Creation Failures

When the task does not run, avoid jumping straight to the task function body. First check whether the task object was created and whether the scheduler started.

Creation resultpdPASS or non-NULL handle?
→
Memory sourceHeap or static buffers valid?
→
Scheduler pathDid scheduler start or return?

Dynamic failure

Heap too small, allocation disabled, or return value ignored.

Static failure

Missing buffer, wrong size, or static allocation disabled.

Runtime failure

Stack too small, bad parameter pointer, or task never blocks.

Debug task creation from object creation outward, not from the print statement inward.
