Diagnostics, Troubleshooting, and Integration

The final module turns FreeRTOS mechanisms into an engineering workflow: classify the symptom, collect the right evidence, test one hypothesis, and make the smallest justified change.

Course: FreeRTOS Lecture
Embedded Systems Laboratory, Pusan National University
Instructor: Prof. Yunju Baek

Learning path

  • Symptom classification before fixes
  • Stack, heap, task list, and runtime statistics
  • Trace views for timing and kernel object history
  • Assertions, hooks, watchdogs, and health monitoring
  • Integration review across tasks, interrupts, memory, and communication objects
First target: do not tune FreeRTOS by guesswork.

Problem Frame

Problem Frame

Intermittent timingRace, starvation, inversion, ISR latency, and queue pressure rarely reproduce on command.
Wrong fix pressureRaising priority, increasing heap, or changing timeout can hide the true failure.
Limited visibilityPlain logs can distort timing and miss kernel object history.
Integration riskFinal systems combine memory, scheduler, interrupt, and communication failures.
A good diagnosis starts by naming the failure class.

Stack and Heap Evidence

Stack and Heap Evidence

Stack evidence

  • High-water mark under worst expected call path.
  • Stack overflow hook enabled in debug builds.
  • Large local arrays reviewed.

Heap evidence

  • Free heap and minimum-ever free heap.
  • Creation failure and malloc failed hook.
  • Fragmentation risk if allocation continues at runtime.

Wrong fix warning

  • Increasing heap does not fix stack overflow.
  • Increasing stack does not fix object leak.
  • Measure before changing memory sizes.
Stack and heap failures need different evidence.

FreeRTOS Module 15
Assertions and Hooks

Assertions and Hooks

configASSERT()

  • Catches illegal assumptions close to the source.
  • Useful for interrupt priority checks.
  • Should record enough location evidence.

Stack overflow hook

  • Detects configured stack overflow checks.
  • Records task identity if possible.
  • Often triggers late unless margins are monitored.

Malloc failed hook

  • Records allocation failure.
  • Connects to heap policy.
  • Should not attempt complex recovery blindly.
Hooks and assertions should leave usable evidence before the system resets.

Health Monitor Task Pattern

Health Monitor Task Pattern

Collect

  • Task stack margins.
  • Minimum-ever heap.
  • Queue/buffer pressure counters.
  • Watchdog feed conditions.

Publish

  • Compact status snapshot.
  • Periodic UART or telemetry record.
  • Fault code stored before reset.

Protect

  • Avoid heavy work in monitor.
  • Do not mask failures by feeding watchdog unconditionally.
  • Keep counters cheap and deterministic.
A health monitor should reveal failures, not hide them.

FreeRTOS Module 15
Wrong Fix Patterns

Wrong Fix Patterns

Priority inflation

  • Raising priority hides queue pressure or long critical sections.
  • System becomes harder to reason about.
  • Fix by tracing wait reason first.

Memory padding

  • Increasing stack or heap without identifying the failing object.
  • Wastes RAM and leaves cause unknown.
  • Fix by measuring stack and heap separately.

Timeout guessing

  • Longer timeout hides deadlock or service delay.
  • Shorter timeout creates noisy failures.
  • Fix by defining recovery policy.
A fix without evidence is a new experiment pretending to be a solution.

Mini Case: Missed UART Command

Mini Case: Missed UART Command

Symptom

  • Command sometimes missing.
  • UART ISR still fires.
  • Parser task appears alive.

Evidence

  • Stream buffer dropped byte count.
  • Parser wake latency.
  • Trigger level and buffer fill history.

Decision

  • Adjust parser priority only if trace supports it.
  • Adjust buffer size only if pressure supports it.
  • Review ISR send return values.
The case links Module 08 and Module 14 evidence.

Mini Case: Random Reset Under Load

Mini Case: High-Priority Task Late

Symptom

  • High-priority worker misses deadline.
  • Increasing priority does not help.
  • CPU load seems moderate.

Evidence

  • Response time trace.
